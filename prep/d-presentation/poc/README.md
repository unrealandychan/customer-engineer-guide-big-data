# POC Overview — Multi-Brand Retail Data Platform

**Purpose:** This POC is the "hands-on keyboard" proof for Interview D. It demonstrates the exact solution described in the presentation — live, runnable, in a real GCP project. This directly addresses the "tech depth gap" feedback from Round A.

---

## What This POC Proves

| Component | What It Demonstrates |
|---|---|
| `schema/` — 3 brand raw schemas | The heterogeneity problem is real. Three different CustomerID structures, order schemas, product taxonomies. |
| `schema/canonical_schema.sql` | The solution: a brand-agnostic canonical schema with `brand_id` + `customer_key` as the unified identity layer. |
| `pipelines/ingest_batch.py` | Batch ingestion: synthetic data generation for all 3 brands + write to GCS. Simulates nightly ERP/CRM loads. |
| `pipelines/transform_canonical.sql` | The identity resolution + canonical transformation: 3 brand schemas → unified `customer_360`. Demo-able live in BigQuery console. |
| `pipelines/streaming_pipeline.py` | Real-time streaming: Pub/Sub → Apache Beam (Dataflow) → BigQuery. Demonstrates exactly-once semantics, late data handling, windowing strategy. |
| `models/demand_forecast_notebook.ipynb` | BigQuery ML ARIMA_PLUS demand forecasting — pure SQL, no separate ML cluster. |
| `dashboard/looker_studio_setup.md` | Looker Studio connected to BigQuery `customer_360` — live cross-brand dashboard in the presentation. |

## Prerequisites

1. **GCP Project:** Create a new GCP project (or use existing). Note the `PROJECT_ID`.
2. **Enable APIs:**
   ```bash
   gcloud services enable bigquery.googleapis.com \
     pubsub.googleapis.com \
     dataflow.googleapis.com \
     storage.googleapis.com \
     notebooks.googleapis.com
   ```
3. **Create GCS Bucket:**
   ```bash
   gsutil mb -l US-CENTRAL1 gs://{PROJECT_ID}-portfolio-raw
   ```
4. **Create BigQuery Datasets:**
   ```bash
   bq mk --dataset {PROJECT_ID}:raw_brand_a
   bq mk --dataset {PROJECT_ID}:raw_brand_b
   bq mk --dataset {PROJECT_ID}:raw_brand_c
   bq mk --dataset {PROJECT_ID}:canonical
   bq mk --dataset {PROJECT_ID}:streaming
   bq mk --dataset {PROJECT_ID}:mart
   bq mk --dataset {PROJECT_ID}:ml
   ```
5. **Install Python dependencies:**
   ```bash
   pip install google-cloud-bigquery google-cloud-storage google-cloud-pubsub \
     apache-beam[gcp] pandas pyarrow faker
   ```

## Demo Sequence (Use This During Presentation)

Run these in order during the "Technical Deep-Dive" section of your presentation:

### Step 1: Show the Problem (2 min)
- Open BigQuery console → show `raw_brand_a.orders`, `raw_brand_b.orders`, `raw_brand_c.orders`
- Highlight: different column names for customer ID, different timestamp formats, different product fields
- "This is why off-the-shelf ETL fails here."

### Step 2: Run the Canonical Transformation (3 min)
- Open BigQuery console → run `transform_canonical.sql` (or show pre-run `canonical.customer_360` table)
- Show a cross-brand customer record: same person in Brand A (cust_id=101) and Brand C (loyalty_number=LYL_888)
- "This is the identity resolution step. A customer who shops at all 3 brands is now one record."

### Step 3: Show Streaming Pipeline (3 min)
- Open terminal → run `python pipelines/streaming_pipeline.py --mode=publisher` (starts publishing POS events)
- Switch to BigQuery console → run live query: `SELECT * FROM streaming.streaming_orders ORDER BY event_time DESC LIMIT 10`
- Refresh every 30 seconds to show new rows appearing
- "This is near-real-time POS data from Brand A's physical stores landing in BigQuery."

### Step 4: Run Demand Forecast (2 min)
- Open the Jupyter notebook → execute the BigQuery ML cells
- Show the `EXPLAIN_FORECAST` output with the forecast chart
- "This demand forecast runs entirely in SQL — no Python model server, no separate ML infrastructure."

### Step 5: Show Dashboard (2 min)
- Open Looker Studio (pre-connected, loaded ahead of time)
- Show the cross-brand sales by day / brand / channel chart
- "This is what the CIO sees: portfolio-level visibility, updated every 15 minutes."

**Total demo time: ~12 minutes** — which fits exactly in Section 3 of the 30-minute presentation.
